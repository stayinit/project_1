# Сервис блокировки платежей клиентов банка
## Исходные данные:
Представь, что ты системный аналитик на проекте в банке. В вашем продукте реализован сервис по отправке платежей юридическим лицам. Но периодически возникают ситуации, когда платежи надо приостановить, потому что клиент ненадежен (есть подозрение на мошенничество и нужно время разобраться), либо клиент предоставил не те реквизиты для перечисления и платежи отбиваются банком клиента (нужно время на выяснение верных реквизитов). 

Запрос бизнеса следующий: 
- нужна возможность заблокировать платежи конкретного клиента 
- нужна возможность разблокировать платежи клиента 
- нужна возможность проверить, заблокирован ли клиент 
- нужна возможность отличать блокировки мошенников от блокировок добропорядочных клиентов

Тебе надо предложить решение по реализации этого запроса бизнеса: 
Подготовить спецификацию (желательно в формате OpenAPI / Swagger) endpoint’ов для реализации задач выше и предложить структуру хранения нужной информации в БД.
## Стек:
REST API, Swagger, PostgreSQL, SQL, UML/Data Modelin
## Методы и подходы:
- Анализ бизнес-требований
- Проектирование REST API
- Реляционное моделирование данных
- Документирование API в стандарте OpenAPI
## Реализация проекта:
1. Разработана спецификация API из 4 endpoints:
- GET /clients/{clientId}/block — проверка текущего статуса клиента
- POST /clients/{clientId}/block — блокировка клиента
- PATCH /clients/{clientId}/block — изменение данных блокировки
- DELETE /clients/{clientId}/block — разблокировка клиента
2. Спроектирована структура БД с таблицами:
- clients — учетные данные и базовые атрибуты клиента
- client_blocks — история блокировок и причины их применения
3. Реализована бизнес-логика:
- разделение блокировок по типам (FRAUD — мошенничество, INCORRECT_DETAILS — некорректные реквизиты)
- поддержка временных блокировок (expiresAt)
- ведение истории блокировок с сохранением причины, инициатора и дат
- предотвращение повторной блокировки уже заблокированного клиента
